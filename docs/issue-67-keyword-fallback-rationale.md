# Issue #67: キーワードフォールバックを戻す理由

## 1. 設計意図（当初の判断）

### 条文の前後で判定する方針

- **rawText**（改正後全文）と **rawTextPrev**（改正前全文）があれば、AI が「今まで無かったリスクは何か」を特定できるはず。
- 改正前と改正後を比較すれば、「新たに発生した」罰則・義務リスクを正確に検出できる。

### キーワードを廃止した理由

- キーワードで引っ掛けると、**元々過料のリスクがあったものも過料フラグが立つ**。
- 例: 既存の過料規定を文言整理しただけの改正でも、「過料」が条文に含まれるため financial と判定される。
- これは**偽陽性**（誤検知）であり、設計意図に反する。
- そのため、キーワード検知を廃止し、**AI の前後文脈による判定のみ**にした（Issue #65）。

---

## 2. 現状: 条文の前後があっても AI がうまくいかない

### 事実

- 全 956 件中、生存・金銭・信用のいずれかがついているのは **4 件 (0.4%)** のみ。
- 風俗営業法（罰金3億円・許可取消）、公益通報者保護法（罰則）、物流２法（CLO 罰金・過料）など、本来リスク該当すべき法令がほぼすべて **other** と判定されている。

### 想定される原因

1. **rawTextPrev が無い場合**
   - rawTextPrev は `amendmentRevisionId` がある場合のみ e-Gov API で取得。取得失敗時は null。
   - 改正前全文が無い場合、プロンプトは「改正後の全文を評価する」と規定しているが、AI が other を返しがち。

2. **「新たに発生した」の判定が厳格すぎる**
   - プロンプト: 「改正前の条文に既にあった規定は含めない」
   - 罰則の**強化**（例: 罰金額の引き上げ）や**明確化**は「新規追加」とみなされにくく、other になりがち。

3. **validateRiskTypeInText の二重チェック**
   - AI が survival/financial/credit を返したとき、条文に該当する規定が「明示されているか」を検証。
   - 検証で false と判断されると other に上書きされる
   - 逆に、AI が other を返した場合は検証されずそのまま採用

4. **AI の出力傾向**
   - 条文が長く複雑な場合、リスク種別の判定を避けて other を返しやすい可能性。

---

## 3. キーワードフォールバックのトレードオフ

### 戻す理由（暫定措置として）

| 観点 | 現状（AI のみ） | キーワードフォールバック追加後 |
|------|-----------------|-------------------------------|
| **偽陰性**（新たにリスクがあるのに検出されない） | 多数（99.6% が other） | 減少 |
| **偽陽性**（元々リスクがあったものもフラグ立つ） | 少ない | 増加する可能性 |

- **現状**: 偽陰性が極端に多く、一覧で「生存・金銭・信用」でフィルタしてもほぼ何も出てこない。
- **キーワードフォールバック**: 偽陽性（元々過料があったものも過料フラグ）のリスクを許容する代わりに、偽陰性を減らす。**暫定措置**としての位置づけ。

### ユーザーが懸念する点（その通り）

- キーワードで引っ掛けると、**元々過料のリスクがあったものも過料フラグが立つ**。
- これは設計意図に反する。本来は「新たに発生した」リスクのみを検出したい。

---

## 4. 結論: 戻す理由を一言で

> **条文の前後がわかれば理論上は特定できるはずだったが、現状 AI が other を返しすぎて「新たに発生したリスク」がほとんど検出されない。そのため、暫定措置としてキーワードフォールバックを追加し、偽陰性を減らす。偽陽性のリスクは残る。**

---

## 5. 根本解決の方向性（キーワードに頼らない）

1. **rawTextPrev の取得率向上**: amendmentRevisionId が無い場合の対応、e-Gov API の取得失敗時のリトライなど。
2. **プロンプトの見直し**: 「罰則の強化・明確化」も「新たに発生した」に含めるか、スコープを調整。
3. **validateRiskTypeInText の見直し**: 検証が厳しすぎる場合、条件を緩和するか、other のときは検証を追加する（逆検証）か。
4. **AI の生出力の分析**: なぜ other を返すのか、rawTextPrev の有無別の傾向を調査。

---

## 6. キーワードフォールバックの適用条件（検討）

偽陽性を減らすため、キーワードフォールバックの適用条件を絞る案:

- **rawTextPrev が無い場合のみ**キーワードフォールバックを適用する。
  - 前後があれば AI に任せる。前後が無い場合のみキーワードで補う。
- これにより「元々過料があったもの」の多くは rawTextPrev があれば AI が other を返す想定で、キーワードは発動しない。

ただし、現状 rawTextPrev の保有率や、rawTextPrev が無い場合の割合は未調査。
