# リスクの定義検討（Issue #16）

「リスク」を**企業に不利益があるもの**として定義し直すための検討メモ。  
親 Issue: #19。実装は #17 で行う（本ドキュメントは定義の確定用）。

---

## 1. リスクの3軸（採用案）

企業にとっての不利益を、次の **3 種類** にシンプルに分ける。

| 種類 | 説明 | イメージ |
|------|------|----------|
| **【生存リスク】** | 業務停止・免許取消 | 会社が終わる |
| **【金銭リスク】** | 罰金・課徴金 | キャッシュが飛ぶ |
| **【信用リスク】** | 社名公表・勧告 | 仕事と人が消える |

- 1 件の法令改正で、複数のリスク種別に当てはまりうる（例: 業務停止 ＋ 課徴金）。
- 一覧・絞り込みでは「生存 / 金銭 / 信用」のいずれか（または複数）でフィルタできるようにする想定。

---

## 2. 検知方法: キーワード vs LLM

### 法令・事務文章の性質

- 硬い事務的な文章では **用語がわりと定型**（「罰金」「業務停止」「免許取消」「課徴金」「社名を公表」「勧告」など）なので、**キーワードだけでも一定の精度は出しやすい**。
- 一方で、**文脈で意味が変わる**ケースがある:
  - 「罰金を**科する**」→ リスクあり
  - 「罰金を**廃止する**」「罰金**に代えて**……」→ リスクの有無・種類が文脈依存
  - 「勧告**に従わない**ときは……」→ 勧告が制裁として機能している
- **否定・除外・条件**が入ると、キーワードだけだと誤検知（偽陽性）や見落とし（偽陰性）が増える。

### キーワードでいけるか・LLM が必要か

| 方式 | 利点 | 欠点 |
|------|------|------|
| **キーワード** | 実装が簡単・コストなし・再現性が高い。法令用語はある程度パターンが決まっているので、**まずはキーワードでカバーするのは現実的**。 | 否定・条件・言い換えに弱い。「廃止」「除く」が近くにあると精度が落ちる。 |
| **LLM** | 文脈・否定・条件を読める。「制裁としての勧告か」「廃止の話か」を判断しやすい。**精度を上げたいなら LLM を検知に使う選択肢はあり**。 | API キー・コスト・レイテンシ・出力のぶれの管理が必要（#12 で検討）。 |

**結論（方針案）**

- **まずはキーワードで 3 軸（生存・金銭・信用）を検知する**。法令文は用語が硬いので、キーワードだけでもそこそこいける想定。
- **「精度が足りない」「否定・文脈で誤検知が多い」と分かったら、#12 で LLM を検知に組み込む**（例: キーワードで候補を出してから LLM でフィルタ／分類、または要約と一緒にリスク種別を LLM に振らせる）。

つまり: **キーワードでスタートし、必要になったら LLM を足す** でよい。

## 3. キーワード案（3 軸用）

検知をキーワードで行う場合の例。必要に応じて増減する。

| 軸 | 想定キーワード例 |
|------|------------------|
| **生存リスク** | 業務停止、営業停止、免許取消、登録取消、許可取消、廃止、取り消し |
| **金銭リスク** | 罰金、課徴金、過料、科料、納付金、徴収 |
| **信用リスク** | 社名公表、氏名公表、公表、勧告、警告、指名、公表する |

- 「取消」「公表」などは文脈でノイズになりうるので、実装時に前後語やルールで調整する想定。
- LLM を導入する場合は、このキーワードを「候補抽出」や「ラベルのヒント」に回すこともできる。

## 4. データの持ち方（#17 実装用のメモ）

- **現状**: `NormChange.penaltyRisk` が HIGH/MID/LOW/NONE の 1 本。
- **3 軸にする場合**:
  - **案A**: 3 本のフラグ（生存リスクあり / 金銭リスクあり / 信用リスクあり）を DB に持つ。絞り込みがシンプル。
  - **案B**: 既存の `penaltyRisk` は残しつつ、`riskTypes` のような JSON や別カラムで「生存,金銭,信用」の複数を持たせる。
- 一覧の「リスクで絞り込み」は、選んだ軸のいずれかに「あり」が付いているものを出す形にすればよい。

---

## 5. このチケット（#16）でやること・やらないこと

- **やること**: 上記「3 軸の定義」と「キーワードでスタート、必要なら LLM」を**定義として確定**する。
- **やらないこと**: 実装（#17, #19 で実装。#12 で LLM を検知に使うかどうかを検討）。

---

## 6. #17 / #19 実装への受け渡し

- **リスクの種類**: 生存リスク / 金銭リスク / 信用リスクの 3 軸。
- **絞り込み**: 各軸ごとに「あり/なし」または「指定した軸のいずれかあり」でフィルタ可能にする。
- **表示**: 一覧・詳細で 3 軸のどれに該当するかを表示（バッジ・アイコン等は #17 で調整）。

以上を #16 の定義案とする。
