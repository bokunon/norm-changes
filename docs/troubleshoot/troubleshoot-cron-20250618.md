# cron が 20250618 ではなく 20250312 から始まった原因調査

## 状況

- **期待**: ローカルで 20250617 まで完了済み → 次は 20250618 のはず
- **実際**: cron ログでは 20250312, 20250313, 20250314 を処理し、4日目で「レスポンスが空です」で失敗

## 調査結果

### 1. タイムライン

| 時刻 (UTC) | 時刻 (JST) | イベント |
|------------|------------|----------|
| 2026-02-25 22:08 | 2026-02-26 07:08 | cron 実行（error、3日処理後失敗） |
| 2026-02-26 01:27 | 2026-02-26 10:27 | Issue #57 デプロイ（空レスポンス → 0件成功扱い） |
| 2026-02-26 22:01 | 2026-02-27 07:01 | cron 実行（ok、0日処理） |

### 2. 原因 A: IngestState の更新タイミング（最有力）

cron が 20250312 から処理したということは、**その実行時点で lastSuccess が 20250311 だった**。

- 現在の本番 API とローカル diagnose の両方で lastSuccess = 20250617 を確認 → **同じ DB を参照**
- したがって、**ローカル refresh で 20250617 に更新したのは、cron (07:08 JST) 実行より後** だったと推測できる
- つまり、cron 実行時点ではまだ lastSuccess が 20250311 のままで、refresh はその後に完了した

### 3. 原因 B: エラー「レスポンスが空です」の発生条件

- **Issue #57 以前**: e-Gov が空レスポンス（その日に公示なし）を返すと `ok: false` でエラー
- **Issue #57 以降**: 空レスポンスは `ok: true, rows: []` で 0 件成功扱い

cron が 2026-02-25 22:08 UTC に実行された時点では、**Issue #57 はまだデプロイ前**（01:27 UTC にデプロイ）。  
そのため、4日目（おそらく 20250315）で e-Gov が空を返した際、旧コードがエラーを返して失敗した。

### 4. 原因 C: DB の分離

ローカルと本番で **別の DATABASE_URL** を使っている場合:

- ローカル refresh → ローカル用 DB の IngestState を 20250617 に更新
- 本番 cron → 本番 DB の IngestState を参照（20250311 のまま）
- 結果として、cron は 20250312 から処理を開始

### 5. 結論

**主因**: 次のいずれか、または両方の組み合わせと考えられる。

1. **タイミング**: ローカル refresh 完了が cron 実行（07:08 JST）より後だった
2. **DB 分離**: ローカルと本番で別 DB を使っており、本番の IngestState が 20250311 のままだった
3. **旧コード**: Issue #57 デプロイ前のため、空レスポンスをエラー扱いして失敗した

### 6. 今後の対策

- **Issue #57 はデプロイ済み**: 空レスポンスは 0 件成功扱いになるため、同種の失敗は発生しにくい
- **次回 cron**: lastSuccess が 20250617 であれば、20250618 から処理される
- **DB 確認**: ローカルと本番で同じ `DATABASE_URL` を使っているか確認する

### 7. 確認方法

```bash
# 本番の IngestState を確認
curl -H "Authorization: Bearer $CRON_SECRET" https://norm-changes.vercel.app/api/ingest/state | jq .

# lastSuccessfulDate が 20250617 なら、次回 cron は 20250618 から
```
