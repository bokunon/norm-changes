# Issue #65: リスク判定の仕様検討 - ソース調査と対応検討

## 1. 問題の整理

### 事例
- **URL**: https://spec-driven-app.vercel.app/norm-changes/cmm31ny3a00aho9u1gtu1ur1p
- **法令**: 労働安全衛生法に係る登録及び指定に関する省令
- **現状の表示**: リスクの種類「その他（手続き変更等）」、penaltyRisk: HIGH
- **penaltyDetail**: 「適合命令・改善命令や**登録取消し・業務停止**の対象となる」
- **問題**: 条文に「登録取消し」「業務停止」が明示されているのに、リスク分類が「その他」になっている。定義上は**生存リスク（事業継続リスク）**が妥当。

---

## 2. リスク関連データの種類と用途（ユーザー認識の整理）

**ユーザーの想定**: 「リスクの定義があって4段階、リスクの内容が何ってサマリが記載されている」= 1種類

**実際の DB / 表示**:

| フィールド | 用途 | 表示場所 | 由来（生成元） |
|------------|------|----------|----------------|
| **riskSurvival / riskFinancial / riskCredit / riskOther** | リスクの**種類**（4分類のラベル） | 詳細「リスクの種類」、一覧のバッジ、フィルタ・絞り込み、通知条件 | AI の `primaryRiskType`（キーワード検知は AI が上書き） |
| **penaltyRisk** (HIGH/MID/LOW/NONE) | リスクの**程度**（4段階） | 詳細「リスク詳細」の表示色（HIGH なら赤）、リスク有無の判定 | AI の `riskLevel` |
| **penaltyDetail** | リスクの**内容のサマリ**（断定文） | 詳細「リスク詳細」の本文 | AI の `penaltyDetailText` |

### 2種類ある理由と意図

| データ | 意図 | プロンプト上のスコープ |
|--------|------|------------------------|
| **リスクの種類**（riskSurvival 等） | フィルタ・絞り込み・通知条件に使う**分類ラベル**。「生存」「金銭」「信用」「その他」のどれか1つ | 「この改正で当てはまる最も厳しいリスク」→ **曖昧**（改正後全文か、新たに発生したか不明） |
| **リスクの内容サマリ**（penaltyDetail） | 人が読む**説明文**。「何がリスクか」を1文で断定 | 「改正後に新たに発生した罰則・義務リスク」→ **明確に「新たに発生した」に限定** |

→ **現状のズレ**: 種類（ラベル）は「改正後全文」ベースで判定されがちな一方、内容サマリは「新たに発生した」に限定されている。2つが別ソースで、整合していない。

## 3. 現行ロジックの流れ（ソース調査結果）

### 3.1 キーワード検知（`src/lib/analyze.ts`）

- 改正後全文（rawText）のみを対象
- キーワードで survival / financial / credit を検知
- **AI の primaryRiskType が返ると、キーワード結果は完全に上書きされる**

### 3.2 AI レポート（`src/lib/report-ai.ts`）

- `primaryRiskType`: プロンプトは「この改正で当てはまる最も厳しいリスク」。スコープは曖昧。
- `penaltyDetailText`: 「改正後に新たに発生した」と明記。
- `validateRiskTypeInText`: survival/financial/credit のときのみ条文検証。other のときは検証しない。

### 3.3 上書きルール（`src/lib/run-analyze.ts`）

- AI の primaryRiskType が存在する場合、キーワード検知結果を**完全に上書き**。

## 4. ユーザー決定事項（2025-02 時点）

### 4.1 「新たに発生した」の定義
- **A（条文に新規追加）**: 含める
- **B（適用条件の変更、例: 任意→必須）**: **必須で含める**
- **C（文言修正のみ）**: 含めない

### 4.2 rawTextPrev が無い場合（新規法案等）
- **案1**: 改正後全文のリスクをそのまま評価（現状維持）

### 4.3 キーワード検知
- 差分ベースは文脈がわからず難しい
- **キーワードは不要**の方向で検討

### 4.4 Issue #65 の案A（言い換え）
- **案A の内容**: AI が「その他」を返したとき、キーワードで「生存」「金銭」「信用」が検知されていれば、キーワードの結果を優先して採用する。
- **言い換え**: 「AI が手続き変更等と判断しても、条文に『登録取消し』『業務停止』などが明示されていれば、生存リスクとして扱う」フォールバック。
- **キーワード廃止の方針との関係**: キーワードを廃止するなら、案A は採用しない。AI の primaryRiskType を信頼する形になる。

### 4.5 プロンプト修正
- ベストだと思う形で修正する（後述の「5. 対応方針」で具体化）

### 4.6 リスクの種類と内容サマリの関係
- 上記「2. リスク関連データの種類と用途」で整理済み。種類（ラベル）と内容サマリのスコープを揃える方向。

### 4.7 優先順位
- ここまでの内容を整理した後に判断。

## 5. 対応方針（整理後）

### 5.1 リスクの種類と内容サマリのスコープを統一

**方針**: 両方とも「改正により新たに発生したリスク」に限定する。

- **primaryRiskType**（→ riskSurvival 等）: プロンプトで「改正により新たに発生したリスク」に限定する旨を明記。
- **penaltyDetailText**（→ penaltyDetail）: 既に「新たに発生した」と明記済み。維持。

### 5.2 キーワード検知の廃止

- 差分ベースは難しいため、キーワード検知は廃止。
- リスクの種類は **AI の primaryRiskType のみ** で決定する。

### 5.3 プロンプト修正（primaryRiskType）

```
primaryRiskType: 改正により新たに発生したリスクのみを評価する。
- 改正前の条文に既にあった規定は含めない。
- 次のいずれかに該当する場合に、該当するリスクを選ぶ:
  (1) 改正で新たに追加された規定（登録取消し、業務停止、罰金等）
  (2) 既存規定の適用条件が変わって実質的にリスクが生じた場合（例: 届出が任意→必須になり、既存の罰則が実効化）
- 文言の微修正のみの場合は other を選ぶ。
```

### 5.4 rawTextPrev が無い場合

- 改正後全文のリスクをそのまま評価（案1）。

### 5.5 Issue #65 の案A

- キーワード廃止に伴い、**案A は採用しない**。
