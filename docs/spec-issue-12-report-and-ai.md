# Issue #12 仕様検討: 企業向けレポートと生成AI・API Key

実装前に「何をどこまでやるか」を決めるための検討メモ。決まったらこのドキュメントを更新し、実装に反映する。

## 1. チケットの範囲（再確認）

- **企業向けレポート**: 取得した法令情報を元に「何をすべきか」をレポートとして出す。
- **生成 AI**: 使うか・使うならどのプロバイダ・何に使うか。
- **API Key**: どこに持つか・誰が設定するか・必須か任意か。

現状: 生成 AI 未使用・API Key 不要。レポート相当は `analyze.ts` のキーワード判定のみ。`rawText` が入るようになったのでキーワードは効くが、「何をすべきか」の自然文説明はまだない。

## 2. 詰める必要があること（一覧）

### 2.1 企業向けレポートの中身

| 項目 | 選択肢・検討メモ | 決定 |
|------|------------------|------|
| **レポートの単位** | 1 法令 1 レポート / 複数法令をまとめた週次・月次レポート / 両方 | **1 法令 1 レポート**。法令をまたいだ話は別途対応。 |
| **含めたい内容** | 例: 法令名・種別・公示日・施行日・「何をすべきか」要約・リスク種別・期限・アクション案・根拠条文 | 2.1 の粒度・表示構成に合わせる（下記）。 |
| **「何をすべきか」の粒度** | 1〜3 文の要約 / 箇条書きのアクション一覧 / 詳細な推奨アクション＋根拠 | **箇条書きのアクション一覧**をサマリとし、その下に**詳細推奨アクション＋根拠**のエリアを設ける。 |
| **表示場所** | 既存の詳細ページに追加 / 別ページ「レポート」/ API のみ / PDF 等エクスポート | **既存の詳細ページ**に記載。現在ある改正前・後全文の**上**にレポートを配置。 |
| **生成タイミング** | 解析（analyze）時に一緒に生成 / 詳細表示時にオンデマンド / バッチで一括 | **アナライズ（analyze）の際に生成**。 |

### 2.2 生成 AI をどう使うか

| 項目 | 選択肢・検討メモ | 決定 |
|------|------------------|------|
| **使うか** | 使う / 当面はキーワード＋テンプレのみで、必要になったら導入 | **使う**。 |
| **使う場合の用途** | 要約・「何をすべきか」文の生成 / リスク分類の補正 / 両方 / その他 | **すべて**：要約・「何をすべきか」（箇条書き＋詳細＋根拠）の生成、リスク分類の補正。 |
| **プロバイダ** | OpenAI / Anthropic / 他（Google 等）/ 抽象化して差し替え可能にする | **推奨案**（下記）。安いモデル＋事前チャージ（予算範囲内）で使えるものを選ぶ。 |
| **入力** | `rawText` のみ / `rawText` + 法令メタ / 前後改正差分（rawTextPrev）も | **推奨案**（下記）。 |
| **出力形式** | 自由文 / 構造化（JSON 等）で DB に保存 | **推奨案**（下記）。 |
| **コスト・レイテンシ** | 1 件あたりの目安、バッチ時の並列数や遅延の許容 | プロバイダ選定に合わせて実装時に設定。 |

#### プロバイダ推奨（安い・内容はしっかり・事前チャージで使いたい）

- **第一候補: OpenAI（gpt-4o-mini）**
  - **理由**: 単価が安く、日本語の要約・構造化出力の品質が安定している。API は従量課金だが、Organization で**月次使用量の上限（Usage limits）**を設定でき、事実上「チャージ範囲内で使う」運用が可能。
  - **事前チャージの扱い**: 厳密なプリペイドではないが、上限を設定しておけばそれを超えて課金されない。クレジットカード登録は必要。
- **代替: Google Gemini 1.5 Flash**
  - **理由**: 入力単価が非常に安い（例: $0.075/1M tokens）。日本語も扱いやすい。Google Cloud の Vertex AI を使えば**プロジェクト予算アラート・上限**で「範囲内で使う」運用がしやすい。
  - 差し替えを想定するなら、**呼び出し層を抽象化**（例: `generateReport(input)` が OpenAI / Gemini のどちらでも動く）にしておくと、後からプロバイダを変えやすい。

#### 入力・出力の推奨案（根拠付き）

- **入力**
  - **推奨**: `rawText`（改正後全文）＋ `rawTextPrev`（改正前全文、あれば）＋ 法令メタ（`title`, `type`, `publishedAt`, `effectiveAt`）。
  - **根拠**: 「何をすべきか」は改正内容に強く依存するため、前後を両方渡すと差分が分かり、アクション抽出の精度が上がる。メタで文脈（種別・施行日など）を渡すと、期限や重要度の記述も一貫しやすい。
- **出力**
  - **推奨**: **構造化 JSON** で DB に保存する。
  - **根拠**: 表示を「箇条書きアクション（サマリ）」「詳細推奨アクション＋根拠」で固定するため、項目ごとに保存するとレンダリングが安定する。再生成や洗い替え時も項目単位で更新しやすい。API の仕様変更やプロンプト変更時も、スキーマを揃えておけば影響範囲を限定できる。
  - **案**: 例として `{ summary: string, actionItems: string[], detailedRecommendations: { action: string, basis: string }[], riskLevel?: string, obligationLevel?: string }` のような形。既存の penaltyRisk / obligationLevel とマージするか「フィルタ用＋詳細」にする方針（2.4）に合わせ、AI 出力にも分類用の項目を含める。

### 2.3 API Key の扱い

| 項目 | 選択肢・検討メモ | 決定 |
|------|------------------|------|
| **保存場所** | 環境変数のみ（例: `OPENAI_API_KEY`）/ Vercel 等の Environment Variables | **環境変数**（例: `OPENAI_API_KEY`）。本番は Vercel 等の Environment Variables に設定。 |
| **必須か** | AI を使う場合のみ必須で、未設定時はキーワードのみにフォールバック / 常に必須 | AI 利用時のみ必須。未設定時はキーワードのみにフォールバック。 |
| **ドキュメント** | `.env.example` に変数名のみ記載・値は書かない / 設定手順を README や docs に記載 | **推奨どおり**：`.env.example` に変数名のみ記載（値は書かない）。必要なら README や docs に設定手順を追記。 |

### 2.4 既存フローとの関係

| 項目 | 検討メモ | 決定 |
|------|----------|------|
| **詳細ページの表示構成** | 種別・法令番号・公示日・施行日・対応重要度・罰則リスク・概要・全文の扱い | **種別・法令番号・公示日・施行日**は今のまま。**対応重要度と罰則リスク**は同じ用途のため、**マージする**か、**フィルタ用の分類＋詳細**の構造に変更。**概要・何をしないといけないか**は 2.1 の内容（箇条書きアクション＋サマリ、下に詳細推奨アクション＋根拠）に差し替え。**改正前・後全文**は一旦そのまま（レポートはその上に配置）。 |
| **analyze との関係** | 現状の NormChange（summary, リスク 3 軸等）はそのまま / レポート用フィールドを追加 / AI で summary 等を上書き | 上記の表示差し替えに合わせ、レポート用フィールドを追加し、AI 出力で summary / アクション一覧 / 詳細＋根拠 等を保存。対応重要度・罰則リスクはマージ or フィルタ用＋詳細に応じてスキーマを調整。 |
| **データ洗い替え** | AI 導入後、既存 NormChange を再解析してレポートを付け直すか・新規のみか | **洗い替えする**（既存も再解析してレポートを付け直す）。 |
| **その他サジェスト** | 実装中に「こういう項目があると良い」が出た場合 | **一旦実装に含め、後で調整**する方針。 |
