# 対象の特定: 公示済み・施行前を拾うためのプラン

チケット登録の前提として、「公示済みで施行前」を対象に拾えているかどうかを整理し、拾えていない場合の仕様変更プランを示します。

---

## 1. 「公示済み・施行前」の定義（このドキュメントでの運用定義）

| 用語 | 意味 |
|------|------|
| **公示済み** | 官報に公布されている（e-Gov の更新法令一覧に載っている時点で、公布日＝PromulgationDate が入っている） |
| **施行前** | 施行日が未到来、または施行日未定。e-Gov API の `EnforcementFlg` では `1`＝未施行。 |

判定に使えるデータ:

- **e-Gov API**: `LawNameListInfo.EnforcementFlg`（0=施行済, 1=未施行）、`EnforcementDate`（施行日 yyyyMMdd）
- **当システム DB**: `NormSource.effectiveAt`（施行日）。`EnforcementFlg` は**保存していない**。

---

## 2. 現状: 拾えているか？

### 結論（短文）

- **データとしては「公示済み・施行前」の法令も取り込まれている**（一覧 API に含まれるため DB に入る）。
- ただし **「施行前だけを対象として扱う」機能はない**（取り込み時の絞り込みも、一覧のフィルタも未実装）。  
  → 「公示済み・施行前を対象として拾う」ためには、仕様変更が必要。

### 詳細

| 観点 | 現状 |
|------|------|
| 取り込み元 | e-Gov「更新法令一覧取得」`updatelawlists/{yyyyMMdd}`。指定日に更新された法令を**全件**取得。 |
| 施行前の識別 | API には `EnforcementFlg`（0/1）があるが、**DB には保存していない**。`effectiveAt`（施行日）のみ保存。 |
| 施行前の判定 | 現状のデータだけでやるなら「施行前 ≒ `effectiveAt == null` または `effectiveAt > 今日`」。施行日未定は `effectiveAt == null` で表現可能。 |
| 取り込み時の絞り込み | なし。一覧に載った法令は**すべて** NormSource に upsert。 |
| 一覧 API のフィルタ | 公示日（from/to）・種別（type）・タグ（tagId）のみ。**「施行前のみ」は未実装**。 |
| 一覧 UI | 公示日 From/To のみ。施行前チェックや施行日での絞り込みはなし。 |

つまり:

- 「公示済み・施行前」の法令は **DB には入り得る**（一覧に含まれるため）。
- しかし **「施行前だけを見る／対象にする」ことができない**ので、**対象の特定**という意味では「拾えていない」とみなすのが妥当。

---

## 3. 拾うための仕様変更の方向性

「公示済み・施行前を対象として拾う」を実現するには、次のいずれか（または組み合わせ）が必要です。

1. **一覧で「施行前のみ」を選べるようにする**  
   → 対象の特定がしやすくなる。データは全件のまま。
2. **取り込み時に「公示済み・施行前のみ」に絞って保存する**  
   → 施行済は DB に残さない。要件次第。
3. **施行前かどうかを DB に明示する**  
   → `EnforcementFlg` を保存するか、`effectiveAt` から算出した「施行前フラグ」を持たせる。フィルタや表示の根拠になる。

推奨は **「1 + 3」**（データは残しつつ、施行前を明示して一覧で絞れるようにする）です。  
「2」は「施行済は一切扱わない」という product 判断がある場合に検討します。

---

## 4. 具体的なプラン（チケット分割案）

### プラン A: 最小限（一覧で施行前だけ見えるようにする）

**目的**: 既存データの `effectiveAt` だけで「施行前」を判定し、一覧 API と UI で「施行前のみ」を選べるようにする。

| # | 内容 | やること |
|---|------|----------|
| A1 | 一覧 API に「施行前のみ」フィルタを追加 | `GET /api/norm-changes` にクエリ `enforcement=not_yet` を追加。`normSource.effectiveAt == null || normSource.effectiveAt > 今日` で絞り込む。 |
| A2 | 一覧 UI に「施行前のみ」を選ぶ手段を追加 | チェックボックスや「施行前」「施行済」「すべて」のラジオなどで `enforcement=not_yet` を付与して API を呼ぶ。 |

- **DB スキーマ変更**: 不要。
- **取り込み処理**: 変更不要。全件取り込みのまま。
- **注意**: `EnforcementFlg` は使わないので、e-Gov の「未施行」と完全一致しない可能性はある（施行日が未定の扱いなど）。必要なら後から B を追加。

---

### プラン B: 施行状態を DB に明示する（推奨）

**目的**: e-Gov の `EnforcementFlg` を保存し、「公示済み・施行前」を確実に識別できるようにする。

| # | 内容 | やること |
|---|------|----------|
| B1 | NormSource に施行状態を保存する | `NormSource` に `enforcementStatus`（例: `ENFORCED` / `NOT_YET` / `UNKNOWN`）を追加。取り込み時に `LawNameListInfo.EnforcementFlg` をマッピング（0→ENFORCED, 1→NOT_YET、なし→UNKNOWN）。既存レコードは NULL または UNKNOWN でマイグレーション。 |
| B2 | 一覧 API で施行状態でフィルタ | `GET /api/norm-changes` に `enforcement=not_yet` 等を追加。`normSource.enforcementStatus === 'NOT_YET'` で絞り込み。`effectiveAt` のみの判定と併用してもよい（後方互換）。 |
| B3 | 一覧 UI で施行前を選べるようにする | プラン A2 と同様。API の `enforcement` パラメータを渡す。 |

- **DB**: マイグレーション 1 本（`enforcementStatus` 追加）。
- **取り込み**: `egov-api` の `lawInfoToNormSourceFields` で `EnforcementFlg` を渡し、ingest で `enforcementStatus` を保存。

---

### プラン C: 取り込み時に「公示済み・施行前のみ」に絞る（任意）

**目的**: 施行済は DB に保存しない。対象を「公示済み・施行前」に限定する product 判断がある場合。

| # | 内容 | やること |
|---|------|----------|
| C1 | 取り込み時に施行前のみ保存 | `GET /api/ingest/laws?date=yyyyMMdd` 内で、`EnforcementFlg === '1'`（または `effectiveAt == null || effectiveAt > 今日`）のものだけ upsert。それ以外はスキップ。 |

- **注意**: 施行済の法令が一切残らなくなる。履歴や「施行済も一覧で見たい」要件があるなら C は採用しない。

---

## 5. 推奨する進め方（チケット登録用）

1. **まず「対象の特定」を完了させる**
   - **プラン B1** … `enforcementStatus` を追加し、取り込みで `EnforcementFlg` を保存する。
   - **プラン B2** … 一覧 API に `enforcement=not_yet` を追加。
   - **プラン B3** … 一覧 UI に「施行前のみ」を選ぶ操作を追加。

2. **チケットの切り方の例**
   - チケット1: 「NormSource に enforcementStatus を追加し、取り込み時に e-Gov EnforcementFlg を保存する」
   - チケット2: 「norm-changes 一覧 API に施行前フィルタ（enforcement=not_yet）を追加する」
   - チケット3: 「法令インパクト一覧画面で『施行前のみ』を選べるようにする」

3. **プラン C** は「施行済を保存しない」という方針が決まった場合だけ別チケットで検討。
