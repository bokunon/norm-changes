## 3. ZIP の中身（20260214 の例）

- **1 本の CSV**: `R080214.csv`  
  - その日に更新された法令の**一覧**（法令種別・法令番号・法令名・公布日・改正法令名・施行日・法令ID・本文URL・未施行 など）。
  - カラム例: `法令種別,法令番号,法令名,法令名読み,旧法令名,公布日,改正法令名,改正法令番号,改正法令公布日,施行日,施行日備考,法令ID,本文URL,未施行`
- **複数ディレクトリ**: 各 1 件の法令に対応。  
  - ディレクトリ名: `{法令ID}_{yyyyMMdd}_{改正ID}`（例: `342AC0000000081_20260214_507AC0000000032`）
  - 中身: 同名の **1 個の XML ファイル**（法令本文の条文データ）

→ **一覧 = CSV。本文 = ディレクトリ内の XML。**  
→ 現行の v1 `updatelawlists` は「一覧だけの単一 XML」。bulkdownload は「一覧 CSV + 法令ごと本文 XML」がセットで取れる。

## 5. 確認結果（方針確定）

1. **取得経路**: **V2（bulkdownload）に切り替え**する。
2. **ZIP の扱い**: bulkdownload を使う場合、ZIP 解凍・CSV パース・必要なら XML 本文の取り込みまでバックエンドで行う → **OK**。
3. **一覧と本文**: 本文があるなら使いたい。実際の内容の確認結果と、以前の全文の要否を以下にまとめる。
4. **公示日**: 「直近の変更の公示日」は CSV の「改正法令公布日」を使う（#15 と一致）。

## 6. bulkdownload の「本文」の中身と、以前の全文の要否

### 6.1 bulkdownload の各 XML は「変更差分」か「全文」か

- 実際に取得した XML（例: 415CO0000000497_20260214_507CO0000000417.xml）は、**Law ルート直下に LawBody があり、Article / Paragraph が並ぶ構造**。
- つまり **その改正を反映した時点の法令の全文** が入っている（変更箇所だけの差分ではない）。
- したがって「法律・省令・政令の全体像」は、**bulkdownload の XML 1 本で掴める**（その改正後時点の全文）。
- ただし **改正前の版** は同 ZIP には含まれない。  
  → 「以前と比較して何が変わったか」を見るには、**改正前の全文を別途取得する必要がある**。

### 6.2 以前の全文は必要か

- **必要**、という認識で整理する。
- 理由は次の 2 点を満たすため:
  1. **規則変更のインパクト**: 「努力義務だったのが罰金対象になった」「今までスルーでよかったが対応しないと業務停止」など、**変更前後の比較**でないと分からない。
  2. **対象者が何をすべきか**: 変更内容を踏まえて「法律の対象者が何をすべきか」を整理するには、**改正前の全文と改正後の全文の両方**があるとよい（差分の特定・要約のため）。
- なので「以前の全文取得」は、**マージするかどうか**というより、**比較用に取得する**ことが目的になる（取得したうえで、新旧を突き合わせて差分・影響を整理する想定）。

### 6.3 以前の全文をどう取るか（履歴 API）

- 履歴を扱うには、**改正履歴一覧** と **特定版の本文取得** の両方が必要。
- 想定している候補は次の 2 種:
  - **`/law_revisions/{law_id_or_num}`** … 法令の改正履歴一覧（どの revision があるか）
  - **`/law_data/{law_id_or_num_or_revision_id}`** … 法令データ（特定の revision の本文）
- **どちらか一方だけでは足りない**想定:
  - `law_revisions`: 「いつ・どの revision があるか」が分かる → その中から「直前の revision」を特定するため。
  - `law_data`: 指定した revision（または法令ID・番号）の**本文**を取得するため。
- 流れのイメージ:
  1. bulkdownload（または一覧 API）で「この日付で更新された法令」とその **law_id** を取得。
  2. **`law_revisions/{law_id}`** でその法令の改正履歴一覧を取得。
  3. 履歴から「今回の改正の直前の revision」を特定。
  4. **`law_data/{revision_id}`**（または v2 のパスに合わせた指定）で **以前の全文** を取得。
  5. 改正後 = bulkdownload の XML、改正前 = law_data で取得した本文、として比較・差分整理。

※ v2 の Swagger（https://laws.e-gov.go.jp/api/2/swagger-ui）で、実際のパス名が `law_revisions` / `law_data` か、別名か（例: `law_revision_list` / `law_full_text`）を確認する必要あり。v1 では `/lawdata/{id}` 等の記載あり。

### 6.4 law_data の「最新」と施行前／施行済の整理（未整理だったところ）

**経緯**

- 以前 **/laws（または法令取得API相当）** で施行前を取ろうとしたが、**施行済みしか取れない**ことが分かり、**bulkdownload を使う方針に変更した**。
- そのため、**リビジョン一覧や law_data が「施行前も返すか」「施行済しか返さないか」**で、設計の挙動が変わる。

**law_data の「revision 指定なし＝最新」で足りない理由**

- 仕様には「revision_id を指定しなければ最新が取れる」とある。
- しかし e-Gov 法令 API の多くの説明では、**取得対象は「施行中の現行法令」に限定**されており、**施行前の法令は API では取得できない**旨の記載がある（v1 は施行前非対応、法令取得API・条文内容取得APIは「施行中の現行法令」のみ、など）。
- つまり **「最新」＝施行済の最新** であり、**施行前の「最新」は law_data では取れない可能性が高い**。施行前を取るために bulk を使う方針にした、という経緯と一致する。
- したがって **revision 指定なしの「最新」だけでは、施行前の版（公示済み施行前）は取れない**想定でよい。

**リビジョン一覧・law_data が施行前を返すか**

- **law_revisions**（改正履歴一覧）や **law_data(revision_id)** が、**施行前の revision も含めて返すか**、**施行済の revision のみか**は、現時点では仕様がはっきり整理できていない。
- 施行前も返すなら、revision 指定で「一つ前の版」や「施行前の版」を直接取りにいける。施行済しか返さないなら、**「一つ前の版」の取得は bulk の ZIP 解凍結果と突き合わせて特定する**必要がある。

**了解した方針**

- **いったんリビジョンで取って、ZIP 解凍して出てきたものと比較して、一つ前の版を特定する**、という意図で了解。
- つまり「bulkdownload の ZIP で得た法令（改正後）を基準に、law_revisions で履歴を取得し、ZIP に含まれる revision と突き合わせて『一つ前の revision』を特定し、必要なら law_data(revision_id) で以前の全文を取る」という流れ。施行前／施行済の API の返却範囲がはっきりしたら、その仕様に合わせて挙動を決める。
